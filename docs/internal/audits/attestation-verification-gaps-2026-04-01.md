# Blockchain Attestation Verification Gaps Audit

**Date:** 2026-04-01
**Issue:** [#19](https://github.com/rickhallett/thepit/issues/19)
**Scope:** All six gaps identified in the original issue
**Auditor:** agent (chore/19-attestation-audit)

## Executive Summary

The attestation implementation is substantially more mature than the original issue
(2026-03-12) anticipated. Several gaps have been partially or fully addressed since
the issue was filed. No real private keys are committed. The TS-to-Go ABI parity
is now verified with golden-value tests. The main remaining exposures are:
the lack of a write-path live chain test (read-path is covered), and the incomplete
`pitnet submit` command (encode-only, no broadcast).

## Gap Analysis

### Gap 1: No Live Chain Integration Test

**Original severity:** HIGH
**Current status:** PARTIALLY ADDRESSED

**Finding:** Live chain integration tests now exist, but only for the read path:

- `pitnet/internal/chain/chain_integration_test.go` (7 tests, `-tags chain`):
  Tests `BlockNumber`, `ChainID`, `GetAttestation`, decode, non-existent UID, schema
  match, and consistency across multiple fetches. All against Base L2 mainnet.
- `pitnet/cmd/proof_integration_test.go` (3 tests, `-tags chain`):
  End-to-end tests that build the pitnet binary and run `pitnet proof` against a
  known mainnet attestation UID (`0x13da22...0724`).

**Remaining gap:** No test creates an attestation on a testnet (Base Sepolia),
reads it back, and confirms the write/read round-trip. The write path
(`lib/eas.ts` `attestAgent()`) is tested only with mocked EAS SDK and ethers.

**Actual risk:** Medium. The read path is proven against real on-chain data. The
write path uses the official EAS SDK which handles transaction construction. The
risk is that a future EAS SDK version changes the transaction format or return
shape, which would only be caught in production.

**Fix cost:** Medium. Requires a funded Base Sepolia wallet, a registered test
schema, and CI infrastructure for testnet tests. The test itself is straightforward
but the infrastructure setup is not trivial. Estimated: 30-45 agent-minutes.

### Gap 2: TS vs Go ABI Encoding Parity

**Original severity:** MEDIUM
**Current status:** ADDRESSED

**Finding:** This gap is closed. The following evidence proves parity:

1. `scripts/eas-abi-golden.ts` generates golden hex values from the real EAS SDK
   `SchemaEncoder.encodeData()` for two test cases (populated fields and empty fields).
2. `pitnet/internal/abi/abi_parity_test.go` contains:
   - `TestEncodeParityWithEasSDK`: Compares Go-encoded bytes against the golden
     hex from the TS EAS SDK. Byte-for-byte match.
   - `TestEncodeParityEmptyFields`: Same comparison with empty optional strings.
   - `TestEncodeDecodeMainnetAttestation`: Round-trips real mainnet attestation
     data through the Go encoder/decoder.
3. `pitnet/internal/abi/crypto_integrity_test.go` adds 10 additional integrity
   scenarios including determinism (1000 iterations), truncation rejection,
   slot position verification, schema string parity, and tamper detection.

**Actual risk:** Low. The parity is verified with golden values generated by the
same EAS SDK used in production. The only residual risk is if someone updates the
TS schema string without updating Go, but `TestIntegritySchemaStringParity` catches
that at Go test time.

**Recommendation:** No further action needed. Consider adding the golden value
generation to CI so it regenerates if the EAS SDK version changes.

### Gap 3: pitnet CLI Commands Untested

**Original severity:** MEDIUM
**Current status:** PARTIALLY ADDRESSED

**Finding:** Test coverage has improved since the issue was filed:

- `pitnet/cmd/proof_test.go`: Unit tests for helper functions (`isZeroUID`,
  `formatUnixUTC`, `formatExpiration`, `displayOrNone`). 13 test cases.
- `pitnet/cmd/submit_test.go`: Tests `RunSubmit` with CLI args and JSON file
  input. 2 test cases.
- `pitnet/cmd/proof_integration_test.go`: Live end-to-end tests for `pitnet proof`.

**Remaining gaps:**
- `pitnet verify` (cmd/verify.go): Zero automated tests.
- `pitnet audit` (cmd/audit.go): Zero automated tests.
- `pitnet status` (cmd/status.go): Zero automated tests.
- `RunSubmit` exits the process on validation errors (`os.Exit(1)`), making
  negative-path unit tests impossible without subprocess execution.

**Actual risk:** Low-Medium. The untested commands (`verify`, `audit`, `status`)
are premium/licensed features. `proof` (the free community verification tool) has
the strongest coverage. The risk is regressions in premium CLI features.

**Fix cost:** Low-Medium. Unit tests for the pure logic in verify/audit/status
are straightforward. Testing CLI exit behavior requires subprocess execution
(as done in proof_integration_test.go). Estimated: 20-30 agent-minutes.

### Gap 4: Signer Key Exposure

**Original severity:** HIGH (Security)
**Current status:** NO EXPOSURE FOUND

**Finding:** Thorough investigation confirms no private key exposure:

1. **No .env files committed:** Only `.env.example` and `.env.test.example`
   exist in the repository. Both are template files with empty values for
   `EAS_SIGNER_PRIVATE_KEY=` (no actual key).

2. **.gitignore coverage:** The root `.gitignore` contains `.env*` on line 37,
   which excludes all `.env` variants (`.env`, `.env.local`, `.env.production`,
   `.env.development`). This is a broad pattern that covers common variants.

3. **Git history clean:** `git log --all --diff-filter=A -- '.env' '.env.local'`
   returns no results. No .env file has ever been committed to any branch.

4. **Code handles key absence correctly:** `lib/eas.ts` `requireEasConfig()`
   throws `'EAS_SIGNER_PRIVATE_KEY is required.'` if the key is missing when
   `EAS_ENABLED=true`. The feature is gated behind `EAS_ENABLED=false` by default.

5. **scripts/create-eas-schema.mjs** reads the key from env vars at runtime,
   never hardcodes it.

**Actual risk:** None at present. The `.env.example` pattern with empty values
is correct practice. The `.gitignore` exclusion is sound.

**Recommendation:** No action needed. If the wallet associated with the signer
key has ever been exposed through other channels (e.g., pasted in a chat, stored
in a CI secret that was leaked), rotation would be warranted -- but that is
outside this audit's scope.

### Gap 5: No E2E Attestation Flow Test

**Original severity:** LOW
**Current status:** STILL OPEN

**Finding:** No Playwright or browser-level test covers the full user journey:
create agent -> agent gets attestation hashes -> see attestation link in UI ->
link resolves to easscan.org.

The URL builder (`lib/attestation-links.ts`) is unit-tested with 4 test cases
covering default URLs, custom bases, template substitution, and path normalization.
But no test verifies the UI renders the link or that clicking it reaches easscan.

**Actual risk:** Low. The URL builder is simple and well-tested. The UI rendering
is a standard React component concern. The easscan.org link target is external
and outside our control.

**Fix cost:** Medium-High. Requires Playwright infrastructure, a running app
instance, and likely a pre-seeded agent with attestation data. The value-to-cost
ratio is low given the simplicity of the URL builder. Estimated: 45-60
agent-minutes for meaningful coverage.

**Recommendation:** Defer. The unit tests on `buildAttestationUrl` are sufficient
for the current risk level. An E2E test here would primarily test that React
renders an anchor tag, which has low marginal value.

### Gap 6: pitnet submit Incomplete

**Original severity:** LOW
**Current status:** CONFIRMED, BY DESIGN

**Finding:** `pitnet/cmd/submit.go` explicitly documents this in its header
comment: "Full on-chain submission (signing and broadcasting a transaction)
requires a signer key and is planned for a future release." The command:

1. Accepts attestation fields via CLI args or JSON file
2. Validates required fields and hash format
3. ABI-encodes the payload using `abi.Encode()`
4. Prints the encoded hex data and a note about needing `EAS_SIGNER_PRIVATE_KEY`

The TS write path (`lib/eas.ts`) handles the actual on-chain submission. The Go
CLI is currently a verification and encoding tool, not a submission tool.

**Actual risk:** Low. The architecture deliberately splits the concern: the
web app (TS) writes attestations; the CLI (Go) reads and verifies them. The CLI
encode-only behavior is useful for manual verification and debugging. Making the
Go CLI also submit would duplicate the TS write path without clear benefit.

**Fix cost:** High. Would require implementing transaction signing, nonce
management, gas estimation, and broadcast in Go. The `chain` package has the RPC
infrastructure but no signing support. Estimated: 60-90 agent-minutes.

**Recommendation:** Defer. The current architecture (TS writes, Go reads) is
sound. Adding Go write capability would create a second code path to maintain
for the same operation. Only revisit if there is a use case for CLI-initiated
attestations (e.g., batch attestation migration).

## Conclusion

The attestation system is in better shape than the issue suggested at filing time.
Two of six gaps are fully closed (ABI parity, key exposure). Two are partially
addressed (live chain tests, CLI coverage). Two remain open but are low priority
and low risk (E2E flow, submit completion).

The highest-value remaining work is a write-path testnet integration test (Gap 1
residual), which would close the last HIGH-priority item. The CLI test coverage
gaps (Gap 3) are a reasonable P2 improvement.
