# Technical Prior Art Survey: Infrastructure and Architecture Patterns for Containerized AI Agent Systems

> Architect report, 2026-03-11. Below-decks research dispatch.
> Task: technical architecture comparison of containerized AI agent systems.
> LLM provenance: generated by Claude, not independently verified against live repos.

## Context: What Midgets Is

For comparison throughout this report, midgets' architecture is:

- **Container**: Debian Bookworm Slim, single Dockerfile, non-root `agent` user (uid 1000)
- **Display**: Xvfb virtual framebuffer (:99, 1280x720x24) + fluxbox window manager
- **GUI automation**: `steer` CLI wrapping xdotool (click/type/hotkey/scroll), scrot (screenshot), wmctrl (window list), xclip (clipboard), tesseract (OCR) - all pixel-level
- **Terminal automation**: `drive` CLI wrapping tmux with sentinel protocol (`__START_<token>` / `__DONE_<token>:$?` markers, poll via capture-pane)
- **Job protocol**: YAML files on a shared Docker volume, atomic acquisition via `os.rename()`, jobrunner in watch or once mode
- **Multi-container**: Docker Compose, N workers from shared image, init container for volume ownership, `--replicas N`
- **Governance enforcement**: Docker mount flags (`-v repo:/opt/repo:ro`), no credential injection, role-specific identity files
- **Cross-model review**: 3 different LLM providers (Gemini/Grok/GPT-4.1) in separate containers, structured YAML output, host-side triangulation
- **Audit trail**: Full action traces per agent (JSON/JSONL), run-metadata.yaml, token-usage.yaml, artifacts directory per run
- **VNC debugging**: Optional x11vnc on port 5900 when `MIDGET_VNC` is set

## 1. Containerized Agent Sandboxes

### Anthropic Computer Use Demo
**Repo**: `anthropics/claude-quickstarts/computer-use-demo`
**Architecture**: Docker container with Xvfb, tightvncserver (VNC on :5900), noVNC (HTTP on :6080), Streamlit UI (:8501), and a combined interface (:8080). Runs Claude models via API with Anthropic-defined computer use tools (`computer`, `str_replace_based_edit_tool`, `bash`). The agent loop runs *inside* the container it controls. Single-session, must be restarted between sessions.

**Display stack**: Xvfb + mutter or xfce4-wm (full desktop environment). Recommended resolution: 1024x768 (XGA). For higher resolutions, they scale screenshots down to XGA and map coordinates back up proportionally.

**Key difference from midgets**: Anthropic's demo uses the model's *native computer use API* (the model returns `computer_use` tool calls with coordinate actions). Midgets wraps xdotool/scrot behind a CLI that *any* model can use via text commands. Anthropic's approach requires Claude's computer-use beta API; midgets' approach is model-agnostic.

**What midgets could learn**: The coordinate scaling approach (screenshot at high-res, downscale for the model, upscale coordinates for clicks) is well-documented and would improve midgets if higher-res displays are needed. Also, noVNC (browser-based VNC) is more accessible for debugging than raw VNC.

### OpenHands (formerly OpenDevin)
**Repo**: `OpenHands/OpenHands` (68.9k stars)
**Architecture**: The most architecturally sophisticated agent sandbox. Has a dedicated `containers/` directory with multiple container definitions. Agent runs in a Docker container, orchestrated by a Python backend. Provides SDK, CLI, Local GUI, Cloud, and Enterprise modes. Agents execute via a sandboxed runtime that intercepts commands.

**Sandbox approach**: OpenHands runs the agent controller *outside* the sandbox and sends commands *into* the container. This is the key architectural difference from Anthropic's demo (where agent loop runs inside the container) and from midgets (where the agent CLI runs inside the container). OpenHands uses a client-server model with the container as the server.

**GUI capabilities**: OpenHands recently added computer use support but is primarily a code agent (bash + file editing). It does not provide a full Xvfb/GUI environment as its primary interface - it uses structured tool calls (bash, file_editor) rather than pixel-level interaction.

**What midgets does differently**: Midgets puts the complete agent (including the LLM CLI like Claude Code, Gemini CLI, Codex) *inside* the container. The agent has a full GUI desktop and terminal. OpenHands keeps the LLM orchestration outside the container. Midgets' approach gives the agent a more "physical" presence - it literally sees a screen and types commands.

**What midgets could learn**: OpenHands' SDK abstraction layer (separating agent logic from sandbox runtime) enables running the same agent code locally, in Docker, or in cloud. This portability layer is something midgets lacks.

### SWE-agent + SWE-ReX
**Repo**: `SWE-agent/SWE-agent` (18.7k stars), `SWE-agent/SWE-ReX` (449 stars)
**Architecture**: SWE-agent is a coding agent; SWE-ReX is its *runtime execution framework*. SWE-ReX is the most relevant comparison for midgets' infrastructure.

**SWE-ReX architecture**: A runtime interface for interacting with sandboxed shell environments. Key features:
- Interact with running shell sessions with reliable command completion detection
- Support interactive CLI tools (ipython, gdb) via shell session management
- Multiple parallel shell sessions
- Backend-agnostic: same agent code works with local execution, Docker containers, AWS remote machines, Modal, or Fargate
- Massively parallel execution (30+ instances simultaneously)

**Terminal protocol comparison**: SWE-ReX's shell session management is functionally equivalent to midgets' `drive` CLI + sentinel protocol. Both solve the same fundamental problem: detecting when a command has finished in a shell session and extracting its output. SWE-ReX likely uses a similar marker-based approach. The difference is that SWE-ReX abstracts this across multiple backends, while midgets' `drive` is tmux-specific.

**GUI capabilities**: Neither SWE-agent nor SWE-ReX provides GUI automation. They are terminal-only. This is the biggest architectural gap between SWE-agent and midgets.

**What midgets does differently**: Midgets adds a full GUI layer (Xvfb + steer) on top of the terminal layer. SWE-agent operates purely through text-based tools. Midgets also adds the governance crew concept (multiple containers with different access levels).

**What midgets could learn**: SWE-ReX's backend abstraction (Docker, Modal, Fargate all look the same to the agent) is excellent engineering. Also, their massively parallel execution model is more mature than midgets' Docker Compose N-worker approach.

### E2B (e2b.dev)
**Architecture**: Cloud-hosted sandboxes - lightweight Linux VMs (not Docker containers). Each sandbox is an isolated VM that boots in ~300ms. Provides SDK in Python and JavaScript. Key features:
- Fast boot (sub-second)
- Pause/resume with full state preservation
- Custom templates (define what environment a sandbox starts with)
- Computer Use support (Desktop sandboxes with Xvfb for GUI)
- Up to 24 hours runtime (Pro), 1 hour (Base)
- API-driven: `Sandbox.create()`, `sandbox.commands.run()`, `sandbox.kill()`

**Sandbox isolation**: E2B uses microVMs (likely Firecracker or similar), not Docker containers. This provides stronger isolation than Docker (separate kernel) at the cost of more overhead. Each sandbox is a full Linux VM.

**GUI support**: E2B has a "Desktop sandbox" product specifically for computer use - Xvfb-based, similar to midgets' approach. They provide this as a specialized template.

**What midgets does differently**: Midgets runs locally on your hardware with Docker. E2B is a cloud service with an API. Midgets gives you full control over the container (mount flags, network, credentials). E2B abstracts the infrastructure away but you lose fine-grained control over mount constraints and network isolation.

**What midgets could learn**: Pause/resume is valuable for long-running agent work. The template system (pre-built environments) is cleaner than midgets' single Dockerfile approach. Sub-second boot times would improve the orchestration loop.

### Open Interpreter
**Repo**: `openinterpreter/open-interpreter` (62.6k stars)
**Architecture**: Runs code *on the host machine* by default. The model generates code, the interpreter executes it locally via `exec()`. Has a Dockerfile but it is for convenience, not for sandboxing. The safety model is "ask user confirmation before executing."

**No real sandbox**: Open Interpreter explicitly says "it can interact with your files and system settings." The Docker support is for packaging, not isolation. It runs code directly on whatever machine it is installed on.

**What midgets does differently**: This is the polar opposite of midgets' approach. Open Interpreter trusts the model with full host access. Midgets enforces constraints through container boundaries. This difference illustrates exactly the thesis midgets is proving: governance-by-prompt (Open Interpreter asks "are you sure?") vs governance-by-infrastructure (midgets mount flags prevent access regardless of what the model reasons).

### Devon, Aider
**Devon**: A coding agent. Uses Docker for development environment but the agent logic runs on the host. No GUI capabilities.
**Aider**: A terminal-based pair programming tool. No containerization, no GUI. Runs directly on host. Excellent at code editing but has no sandbox concept.

**Neither provides containerized GUI environments.**

## 2. GUI Automation Stacks for AI Agents

### Approach Taxonomy

| Approach | Projects Using It | Mechanism | Strengths | Weaknesses |
|----------|------------------|-----------|-----------|------------|
| **Pixel-level (xdotool+scrot)** | Midgets, Anthropic CU demo | Screenshot pixels + coordinate clicks | Model-agnostic, works with any app | Fragile to UI changes, OCR errors, resolution-dependent |
| **Vision API (native)** | Anthropic CU, Claude computer use | Model sees screenshot as image, returns coordinates | No OCR needed, model understands context | Requires vision-capable model, high token cost |
| **DOM/Accessibility tree** | Playwright-based agents, WebVoyager | Structured tree of UI elements | Precise, stable selectors | Only works for web/AT-SPI2 apps |
| **Playwright/Puppeteer** | BrowserUse, AgentQ, WebArena | Chrome DevTools Protocol | Fast, reliable for web | Web-only |
| **Selenium** | Older automation frameworks | WebDriver protocol | Broad browser support | Slow, flaky |

### Midgets' Position

Midgets uses pixel-level (xdotool+scrot+tesseract) for GUI and tmux+sentinel for terminal. The SPEC.md explicitly states "No accessibility tree / AT-SPI2 integration (future phase)."

**The DOM vs pixel tradeoff is real**: DOM-level interaction (Playwright, accessibility tree) gives you structured elements with stable selectors. Pixel-level gives you universality (works for any GUI app, not just browsers). For governance review work, midgets' agents primarily need terminal access and occasional browser access. The pixel approach works for the proof but would be fragile at scale.

**Research finding**: The Anthropic computer use documentation recommends XGA (1024x768) resolution to avoid image resizing artifacts. They explicitly state that relying on API-side image resizing "will result in lower model accuracy and slower performance." This is directly relevant to midgets' 1280x720 resolution choice.

### What Midgets Could Learn

1. **Resolution matters**: Consider downsizing screenshots to 1024x768 before sending to models (Anthropic's recommendation)
2. **Accessibility tree as future upgrade**: AT-SPI2 is already scoped out in SPEC.md. Projects like WebVoyager show that structured element trees dramatically improve reliability for repetitive GUI tasks
3. **Hybrid approach**: Use pixel-level for general GUI, Playwright/CDP for web-specific tasks. The Chrome instance is already in the container

## 3. Multi-Model Orchestration for Review/Verification

### Existing Approaches

**Midgets' approach**: 3 different LLM providers (Gemini 2.5 Flash, Grok 3 Mini Fast, GPT-4.1 Mini) each in their own container, each with role-specific prompts (watchdog/weaver/sentinel), each producing structured YAML. Claude excluded from reviewing because it wrote the code under review. Host-side triangulation via verdict counting. This is *actual* multi-model orchestration - different weights, different training data, different priors.

**CrewAI**: Multi-agent framework with "Crews" (teams of agents) and "Flows" (event-driven workflows). Key architectural difference: CrewAI agents all share the same process, same context window, and typically the same model. When CrewAI says "multi-agent," it means multiple *role prompts* on the same model. There is no container isolation between agents. No infrastructure-enforced constraints. This is governance-by-prompt, not governance-by-infrastructure.

**AutoGen (Microsoft)**: Multi-agent conversation framework. Agents are Python objects that exchange messages. No containerization. All agents typically run in the same process and can use the same or different models. No structured review/triangulation protocol.

**LangGraph**: State machine for agent workflows. Does not prescribe multi-model patterns. Single model per node is typical.

**Academic research on cross-model consensus**: The N-version programming literature (from traditional software engineering) establishes that independent implementations catching independent faults is the basis of fault tolerance. The same principle applies: if 3 different models with different training data independently identify the same bug, confidence is higher than a single model's assertion. Midgets' SD-318 (darkcat-alley) codifies this.

### What Midgets Does Differently

Midgets appears to be unique in combining:
1. **Different vendors per reviewer** (not just different model sizes from the same vendor)
2. **Container isolation per reviewer** (each reviewer runs in its own Docker container)
3. **Infrastructure-enforced access control** (reviewers get read-only mounts)
4. **Exclusion of the authoring model from review** (Claude wrote the code; Claude does not review it)
5. **Structured verdict protocol** (YAML schema, not free-text)
6. **Host-side triangulation** (not model-mediated synthesis)

Most multi-agent frameworks (CrewAI, AutoGen, LangGraph) run all agents in the same process with the same model and no isolation. Midgets' approach is closer to N-version programming from safety-critical systems engineering.

### What Midgets Could Learn

- CrewAI's "Flows" concept (event-driven workflow orchestration) could formalize the orchestrator's sequencing logic
- Token/cost tracking could be standardized across providers (midgets already does this in `orchestrate.sh`)
- Weighted voting (some models/roles more trustworthy for certain defect types) rather than simple majority counting

## 4. Infrastructure-Enforced Governance for AI Agents

### The Central Thesis

This is where midgets makes its most distinctive contribution. The SPEC.md states:

> "The container boundary IS the governance boundary. This is the load-bearing design decision."

### Prior Art Comparison

| Project | Sandbox? | Mount constraints? | Network isolation? | Credential isolation? | Governance level |
|---------|----------|-------------------|-------------------|-----------------------|-----------------|
| **Midgets** | Docker container | Yes (`:ro` flags) | Possible (`--network none`) | Yes (per-container env vars) | Infrastructure |
| **Anthropic CU** | Docker container | No | No | No | Prompt + human approval |
| **OpenHands** | Docker container | Partial (workspace mount) | No documented | No | Prompt + sandbox |
| **SWE-ReX** | Docker/Modal/Fargate | Not governance-focused | Not documented | Not documented | Sandbox only |
| **E2B** | MicroVM | API-controlled | API-controlled | API-controlled | Cloud platform |
| **Open Interpreter** | None by default | N/A | N/A | N/A | Human confirmation |
| **CrewAI** | None | N/A | N/A | N/A | Prompt only |

### Analysis

No other project in this survey uses Docker mount flags as a *governance mechanism* - that is, using `:ro` mounts to enforce that a specific agent role (e.g., Watchdog) cannot modify source code regardless of what it reasons. Most projects use containers purely for *isolation* (preventing the agent from affecting the host), not for *role-based access control* within a multi-agent system.

**Docker security profiles**: Docker supports seccomp profiles (syscall filtering), AppArmor profiles (mandatory access control), and capability dropping (`--cap-drop ALL`). None of the surveyed projects document using these for agent governance. Midgets does not use them either but could layer them on top of mount constraints for defense in depth.

**The prompt-vs-infrastructure distinction**: This is well-established in security research. Prompt-based constraints are analogous to "please do not access this file" - they rely on the model's compliance. Infrastructure constraints are analogous to filesystem permissions - they are enforced by the kernel regardless of the model's intent. The security research consensus is clear: enforcement mechanisms must be at a level the agent cannot circumvent. Prompt-level constraints are necessary but not sufficient.

### What Midgets Could Learn

1. **Seccomp profiles**: Drop unnecessary syscalls (e.g., the sentinel agent has no reason to call `mount`, `ptrace`, `setuid`)
2. **Network policy**: `--network none` for reviewers who do not need internet access (currently noted as "future" in SPEC.md)
3. **Resource limits**: `--cpus` and `--memory` are already used in `orchestrate.sh` but could be formalized per role
4. **Read-only rootfs**: `--read-only` flag on Docker run would prevent agents from modifying anything outside explicit mount points
5. **User namespace remapping**: Map container root to unprivileged host user for additional isolation layer

## 5. Agent Steering/Interruption Mechanisms

### How Existing Projects Handle Human Intervention

| Project | Intervention mechanism | Granularity |
|---------|----------------------|-------------|
| **Open Interpreter** | Confirmation prompt before each code execution | Per-action |
| **OpenHands** | Web UI with approval buttons | Per-action |
| **SWE-agent** | CLI flags for auto-approval vs interactive | Run-level |
| **CrewAI** | `human_input=True` on tasks | Per-task |
| **Midgets** | Two mechanisms: infrastructure pipe (steer) + file-based signal | Mid-flight |
| **Anthropic CU** | Streamlit UI with chat interface | Per-conversation |

### Midgets' Approach

Midgets has two steering mechanisms:

1. **Infrastructure steering (steer-watcher pipe)**: Writes stream-json message files into container, steer-watcher daemon delivers to Claude Code via named pipe. Guaranteed delivery if daemon running. Claude Code specific.

2. **Prompt-based steering (signal)**: Appends timestamped instruction to instructions.md. Agent's system prompt tells it to check periodically. Best-effort, model-agnostic.

### What Exists in the Wild

1. **Signal-based**: Unix signals (SIGTERM, SIGINT) to running containers. Docker supports `docker stop` (SIGTERM then SIGKILL after grace period). This is coarse-grained.

2. **File-based**: A "stop file" or "pause file" that the agent checks periodically. Simple but requires agent cooperation (governance-by-prompt again).

3. **API-based**: A REST endpoint on the agent that accepts interrupt commands. OpenHands does this with its server architecture.

4. **Stream-based**: Stdin piping into the container. The agent reads stdin for interrupt signals. This is how interactive terminals work.

### What Midgets Could Learn

- A lightweight HTTP health/control endpoint per container (beyond just job files) would enable finer-grained steering without breaking the job protocol
- The `docker stop` signal chain (SIGTERM -> grace period -> SIGKILL) should be explicitly documented as the interrupt mechanism
- Consider a "checkpoint" pattern: agent periodically writes state, so an interrupted agent can resume from checkpoint rather than restarting from scratch

## 6. Job Protocols for Multi-Agent Coordination

### Comparison of Approaches

| Approach | Projects | Latency | Complexity | Failure handling |
|----------|----------|---------|------------|-----------------|
| **Filesystem YAML** | Midgets | 500ms poll | Low | Atomic rename for contention |
| **Message broker** | None surveyed | ~ms | High (requires broker service) | At-least-once delivery |
| **HTTP API** | OpenHands, E2B | ~ms | Medium | Request/response pattern |
| **In-process** | CrewAI, AutoGen | ~0 | Low | Exception propagation |
| **gRPC** | None surveyed for agents | ~ms | High | Structured errors |

### Midgets' Filesystem YAML Protocol

Midgets uses a filesystem-based job queue:
1. Orchestrator writes job YAML to `incoming/`
2. Worker renames to `.processing` (atomic on Linux via `os.rename()`)
3. Worker executes, writes result YAML to `done/`
4. Orchestrator reads `done/`

This is effectively a *poor man's message queue* implemented via filesystem operations. It has genuinely good properties:
- **Atomic acquisition**: `os.rename()` on Linux is atomic; concurrent workers cannot grab the same job
- **Crash recovery**: `.processing` files indicate in-flight jobs that crashed
- **No additional services**: No Redis, no RabbitMQ, no additional container
- **Human-readable**: YAML files can be inspected by operators

**Limitations**:
- Poll-based (500ms interval) introduces latency
- No ordering guarantees beyond filesystem glob order
- No retry mechanism (a crashed job stays in `.processing` forever)
- No priority queue
- Volume ownership requires init container workaround

### What Midgets Could Learn

- **inotifywait** instead of polling: Linux inotify can notify on file creation, eliminating the 500ms poll latency
- **Retry with dead-letter**: Move `.processing` files older than timeout to a `failed/` directory
- **Consider Unix domain sockets**: Still local, still simple, but event-driven instead of poll-based. Would replace filesystem polling with socket communication while keeping the "no external services" property.

## 7. Agent Observation and Audit Trails

### What Exists

| Project | Action logging | Token tracking | Screenshot recording | Structured traces |
|---------|---------------|---------------|---------------------|-------------------|
| **Midgets** | Full agent traces (JSON/JSONL), run-metadata.yaml, token-usage.yaml | Yes, parsed from provider-specific trace formats | Steer snapshots stored in /tmp/steer/ | Yes (YAML) |
| **OpenHands** | Trajectory files (JSON) | Via LiteLLM | Screenshot history in UI | Yes |
| **SWE-agent** | Trajectory files in `trajectories/` | Yes | No (terminal-only) | Yes (JSON) |
| **CrewAI** | Telemetry (anonymous, opt-in), "Crew Control Plane" for enterprise tracing | Basic via LLM providers | No | Enterprise only |
| **Anthropic CU** | Streamlit UI shows conversation history | Via API response | Screenshots in conversation | No structured audit |
| **E2B** | API-level logging | Via API | No built-in | No |

### Midgets' Audit Architecture

Midgets' audit trail is surprisingly thorough for a proof-of-concept:

1. **Per-agent traces**: Gemini produces JSON with `.stats.models.*.tokens`, Codex produces JSONL with `turn.completed` events, Grok produces raw API JSON responses. The `orchestrate.sh` script parses all three formats and produces a unified `token-usage.yaml`.

2. **Run metadata**: `run-metadata.yaml` captures: run ID, date, image, injected defect description, crew members (role, model, provider, trace file, stderr file), a note that Claude was excluded and why, and the triangulated verdict.

3. **Artifact persistence**: Before Docker volumes are cleaned up, all artifacts are tar-extracted to `data/alley/<run-id>/` on the host.

4. **Review artifacts**: Structured YAML reviews from each agent, prompts preserved, source code preserved for provenance.

### What Midgets Does Differently

- **Provenance chain**: The injected defect is documented in metadata alongside the agent's findings, creating a verifiable chain from "what was planted" to "what was found"
- **Cross-format token parsing**: Handles three different trace formats (Gemini, xAI, Codex) and normalizes to a single schema
- **Explicit confound documentation**: The audit trail documents not just what happened but what confounds were controlled for (e.g., "Claude excluded - it wrote the code under review")

### What Midgets Could Learn

- **OpenTelemetry**: Structured tracing with spans, events, and attributes would make traces queryable. Currently midgets has bespoke parsers for each provider format
- **Screenshot recording**: Periodically capture screenshots during agent execution (not just on `steer see` calls) to create a visual audit trail. Some computer use research projects record full screen captures for replay
- **Structured event log**: Move from per-run artifacts to a centralized event stream that spans runs. The `events.yaml` file in `docs/internal/` is a start but is currently manual

## Summary: Where Midgets Sits in the Landscape

### Genuinely Novel in This Survey

1. **Infrastructure-enforced role-based access control for AI agents** (mount flags as governance, not just isolation)
2. **Cross-vendor multi-model review with explicit authoring-model exclusion** (not same-model "multi-agent")
3. **Structured defect injection + audit trail for proving governance effectiveness** (most projects demonstrate capability, not governance)
4. **Unified container that includes LLM CLI, GUI desktop, terminal automation, and job protocol** (most projects pick one or two of these)

### Known and Accepted Gaps

1. No cloud backend abstraction (SWE-ReX, E2B are ahead here)
2. No accessibility tree / AT-SPI2 (explicitly deferred in SPEC.md)
3. No pause/resume (E2B has this)
4. No real-time interruption mechanism beyond file-based signal
5. Poll-based job queue (could use inotify)
6. No seccomp/AppArmor profiles
7. Single host only (no Kubernetes, no multi-host)

### Top 5 Architectural Learnings to Consider

1. **SWE-ReX's backend abstraction**: If midgets ever needs to run on cloud infrastructure, the same agent code should work unchanged. SWE-ReX solved this cleanly.

2. **Anthropic's resolution scaling**: Downscale screenshots to 1024x768 before sending to models. This is a cheap improvement to accuracy.

3. **noVNC instead of raw VNC**: Browser-based VNC access for debugging is more accessible. Already in Anthropic's demo container.

4. **inotifywait instead of polling**: Replace the 500ms poll loop with Linux inotify for job detection. Strictly better with no added complexity.

5. **Docker `--read-only` + explicit tmpfs mounts**: Strengthen the governance boundary by making the entire rootfs read-only and explicitly mounting only the directories the agent needs to write.
