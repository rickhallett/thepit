## The audit

I grepped my project for blockquotes. 238 hits. Most of them are doing real work - provenance metadata, attributed quotes from transcripts, instructions that define agent behaviour. But some of them are the kind of thing an LLM puts at the top of a file to sound important.

Six were pure slop:

- "Governance is inescapable. This is not reduced governance - it is refined governance."
- "Every change runs the gauntlet before it enters the codebase."
- "When the instruments fail, navigate from last known fixed position."
- "The naval metaphor was scaffolding. The principles underneath are substrate-neutral."
- "Every ship keeps a log. Not for the admiralty, not for posterity, but because the sea does not care what you remember."

All generated by LLM agents. All sitting at the top of a document in a `>` blockquote. All performing seriousness without saying anything the title and first paragraph don't already say.

## The classification

The distinguishing question: does this blockquote contain information that isn't already in the document, or is it decoration?

- **KEEP**: Factual metadata ("Date: 2026-03-08, Status: Implemented"). Attributed quotes with timestamps. Functional instructions ("Read the latest Lexicon on load").
- **KILL**: Unattributed aphorisms. Epigrammatic closures. Sentences that could be deleted without the document losing any information.
- **TRIM**: Mixed - functional content wrapped in decorative phrasing. Keep the instruction, drop the poetry.

I presented the 16 candidates in a muster table - numbered rows, one binary decision each. The Operator accepted all 16 defaults without a single override.

## The tool

The slop attacks will continue. So I built `slopmop` - a CLI tool that scans markdown for slop candidates, classifies them via LLM, and presents them for keyboard-first decisioning.

```bash
slopmop scan .                          # regex scan, no LLM, no cost
slopmop review .                        # scan + haiku classify + interactive muster
slopmop report . --pattern blockquote   # non-interactive report
```

Three phases:
1. **Regex scan** - find blockquotes, em-dashes, emoji, epigrammatic patterns
2. **LLM classify** - claude-haiku-4-5 classifies each candidate as KILL/TRIM/KEEP/SKIP
3. **Interactive muster** - one keypress per finding (k/t/s/Enter/q)

The interesting design question was whether structural heuristics (sentence length, paragraph position, POS patterns) could replace the LLM pass. The answer from prior empirical work: no. Structural heuristics match the surface shape of slop but cannot distinguish good short sentences from bad ones. Each epigrammatic closure is "individually defensible" - that's precisely why a heuristic can't tell them apart.

The architecture collapses to two tiers:
- **T0 (deterministic)**: byte matching, known phrases, zero false positives
- **T2 (semantic)**: needs a model, no shortcut

There is no cheap reliable middle.

## The pattern

These blockquotes show up at the top of documents because that's where LLMs put their "mission statement" when creating a file. Short, punchy, abstract nouns, performs authority. The slopodar taxonomy calls this epigrammatic closure.

The test is simple: delete the blockquote and see if the document gets worse. If it doesn't, it was slop.
